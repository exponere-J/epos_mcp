## What This Cycle Does

Cycle A is a closed diagnostic loop that takes any software system, workflow, or configuration stack and runs it through six structured phases. The output is a set of verified improvements and a permanent record of what was learned — patterns that make the next cycle faster and more accurate.

The cycle never acts on assumption. Every phase produces a file. Every file is the proof. No file = phase not complete.

---

## When Cycle A Runs

- A new client system needs a Sovereign Alignment audit
- EPOS detects repeated failures in its own infrastructure
- Friday (Gemini) identifies cross-cycle signals suggesting internal system weakness
- Jamie manually triggers a diagnostic on any target system
- A 90-day contract milestone requires a verification pass

## Cycle A — Phase-by-Phase Specification

### Phase 1: Scout — Deconstruct

**What this phase IS:**  
A read-only scan of the target system. Scout builds a factual map of what exists — nothing more.

**What this phase IS NOT:**  
Scout does not rank issues. Scout does not suggest fixes. Scout does not make any judgments about severity or importance. Any Scout output containing recommendations is malformed and must be rejected.

**Model:** `TTLG_SCOUT_A` (default: `zhipu-ai/glm-4.5-air:free`)

**Inputs required before phase starts:**
- Target directory path or log file path
- Scope definition: which systems, components, or files are in scope for this run
- Run ID (generated by Auto-Router)

**System prompt for Scout:**
```
You are a data mapper. You do not fix things. You do not rank things. You read the provided code, configuration files, and logs. You output a structured list of: all components found, all dependencies between components, all explicit errors or warnings found in logs, and any structural patterns that appear more than once. Do not explain, recommend, or judge. Output only facts about what is present.
```

**Output artifact:** `Scout_Map.json`  
**Output location:** `vault/runs/{run_id}/Scout_Map.json`

**Scout_Map.json schema:**
```json
{
  "run_id": "string",
  "cycle": "A",
  "phase": 1,
  "target_scope": "string — description of what was scanned",
  "scanned_at": "ISO 8601 timestamp",
  "components": [
    {
      "name": "string",
      "type": "string — file | service | config | module | endpoint",
      "path": "string — full Windows path",
      "dependencies": ["list of component names this depends on"],
      "status": "string — present | missing | partial"
    }
  ],
  "explicit_errors": [
    {
      "source": "string — which file or log",
      "error_text": "string — exact error message",
      "line_number": "integer or null"
    }
  ],
  "repeated_patterns": [
    {
      "pattern_description": "string",
      "occurrences": "integer",
      "locations": ["list of paths where pattern appears"]
    }
  ],
  "coverage_notes": "string — anything Scout could not read and why"
}
```

**Phase 1 complete when:**
- `Scout_Map.json` exists at `vault/runs/{run_id}/Scout_Map.json`
- File is valid JSON
- `components` array has at least one entry
- `scanned_at` timestamp is within the last 60 minutes

**Halt condition:**  
If the target directory does not exist or is unreadable, Scout writes a `FailureArtifact` and raises `ScoutAccessError`. Phase does not continue. Friday is notified.

### Phase 2: Thinker — Evaluate

**What this phase IS:**  
Constitutional evaluation. Thinker compares what Scout found against the EPOS Constitution and the accumulated pattern library. It ranks every issue by severity and projects business impact at 90, 180, and 365 days.

**What this phase IS NOT:**  
Thinker does not propose fixes. Thinker does not write code. Thinker only judges and ranks. Surgeon owns all fixes.

**Model:** `TTLG_THINKER_A` (fixed: `deepseek/deepseek-r1:free` — cannot be changed by PRIORITY_MODE)

**Inputs required before phase starts:**
- `Scout_Map.json` (Phase 1 output)
- `vault/constitution/EPOS_CONSTITUTION_v3.1.md`
- `vault/patterns/` — all existing pattern files

**System prompt for Thinker:**
```
You are an evaluator. You have received a structured map of a software system and a set of constitutional rules and known patterns. Compare each item in the map against the rules and patterns. For every violation or risk you find, assign a severity (Critical | High | Medium | Low) and estimate the business impact if left unaddressed at 90 days, 180 days, and 365 days. Do not propose fixes. Output only a ranked list of issues with evidence from the map.
```

**Output artifact:** `Heal_List.json`  
**Output location:** `vault/runs/{run_id}/Heal_List.json`

**Heal_List.json schema:**
```json
{
  "run_id": "string",
  "cycle": "A",
  "phase": 2,
  "created_at": "ISO 8601 timestamp",
  "item_count": "integer",
  "items": [
    {
      "item_id": "string — HEAL-{run_id}-{sequence}",
      "description": "string — plain language description of the issue",
      "evidence": "string — what in Scout_Map supports this finding",
      "severity": "Critical | High | Medium | Low",
      "constitutional_violation": "string — which rule is violated, or null",
      "pattern_match": "string — matching pattern ID from library, or null",
      "business_impact_90d": "string — what goes wrong in 90 days if not fixed",
      "business_impact_180d": "string",
      "business_impact_365d": "string"
    }
  ]
}
```

**Phase 2 complete when:**
- `Heal_List.json` exists and is valid JSON
- Every item has a severity and at least one business_impact field
- Every Critical item has a `constitutional_violation` or `evidence` field populated

**Halt condition:**  
If `Scout_Map.json` is missing or malformed, Thinker writes a `FailureArtifact` and raises `MissingInputArtifactError`. Friday is notified.

### Phase 3: Governance Gate — Human Approval

**What this phase IS:**  
Human sovereignty. Jamie reviews every item in the Heal List and assigns an explicit status to each. No model makes this decision.

**What this phase IS NOT:**  
This phase cannot be automated. It cannot be defaulted. It cannot be bypassed. If the Governance Gate is empty of approvals, the cycle halts cleanly — no downstream phases run.

**Model:** None — human decision only via Streamlit UI or CLI

**Inputs required before phase starts:**
- `Heal_List.json` (Phase 2 output)

**UI behavior:**
- Displays each item with: description, severity badge (color-coded), evidence, constitutional violation if present, all three business impact projections
- Three buttons per item: **Approve** / **Reject** / **Defer**
- No item may have no status selected — form prevents submission until all items have a status
- On submit: writes `Approved_Heal_List.json` to vault immediately

**Output artifact:** `Approved_Heal_List.json`  
**Output location:** `vault/runs/{run_id}/Approved_Heal_List.json`

**Approved_Heal_List.json schema:**
```json
{
  "run_id": "string",
  "cycle": "A",
  "phase": 3,
  "reviewed_at": "ISO 8601 timestamp",
  "approved_by": "human_steward",
  "approved_count": "integer",
  "rejected_count": "integer",
  "deferred_count": "integer",
  "items": [
    {
      "item_id": "string — matches Heal_List item_id",
      "description": "string",
      "severity": "Critical | High | Medium | Low",
      "status": "approved | rejected | deferred",
      "steward_note": "string or null — optional note from Jamie"
    }
  ]
}
```

**Phase 3 complete when:**
- `Approved_Heal_List.json` exists and is valid JSON
- Every item from `Heal_List.json` appears with an explicit status
- `approved_count` + `rejected_count` + `deferred_count` = `item_count` from Heal_List

**Clean halt condition:**  
If `approved_count` is 0, the system writes `GOVERNANCE_SKIP_{run_id}.json` to `vault/failures/` with reason `all_items_rejected_or_deferred`. Phases 4, 5, and 6 do not run. Friday logs this event. This is not a failure — it is a valid outcome.

### Phase 4: Surgeon — Precision Patching

**What this phase IS:**  
Patch generation for approved items only. Surgeon proposes exact, minimal changes — code diffs, config edits, or process modifications — for every item in the Approved Heal List. Nothing outside the approved scope is touched.

**What this phase IS NOT:**  
Surgeon does not validate its own patches. That is Analyst's domain. Surgeon does not decide what to fix — that was decided at the Governance Gate.

**Model:** `TTLG_SURGEON_A` / `CLAUDE_SURGEON_BACKEND` (default: `deepseek/deepseek-coder-v2-lite:free` via Claude Code orchestration)

**Inputs required before phase starts:**
- `Approved_Heal_List.json` (Phase 3 output) — `approved_count` must be > 0
- Full code context for every file referenced in approved items

**System prompt for Surgeon:**
```
You are a code modifier. You only look at the items in the Approved list. For each approved item, write the exact code, configuration change, or process modification needed to fix only that specific item. Do not touch any code not directly related to an approved item. Do not add improvements, refactors, or optimizations beyond what the approved item requires. Output each change as a named patch with the target file path and the exact replacement content.
```

**Output artifact:** `Proposed_Patches.txt`  
**Output location:** `vault/runs/{run_id}/Proposed_Patches.txt`

**Proposed_Patches.txt structure:**
```
=== PATCH: {item_id} ===
Target file: C:\full\windows\path\to\file.py
Change type: replace | insert | delete | config_edit
Description: One sentence explaining what this patch does

--- BEFORE ---
[exact original code block]

--- AFTER ---
[exact replacement code block]

=== END PATCH ===
```

**Phase 4 complete when:**
- `Proposed_Patches.txt` exists and is non-empty
- Every approved item from `Approved_Heal_List.json` has a corresponding patch section
- No patch section references a file path not in the approved items' scope

**Halt condition:**  
If Surgeon generates patches for items not in the Approved Heal List, those patches are discarded. If all generated patches are out of scope, Surgeon raises `SurgeonScopeViolationError` and halts. Friday is notified.

### Phase 5: Analyst — Sandbox Verification

**What this phase IS:**  
Verification in a safe copy of the target environment. Analyst applies each patch to a sandbox copy, runs checks, and produces a binary PASS or FAIL verdict per patch with supporting evidence.

**What this phase IS NOT:**  
Analyst does not suggest improvements to patches. Analyst does not retry failed patches. Analyst records verdicts and metrics — Surgeon owns any re-patching.

**Model:** `TTLG_ANALYST_A` (default: `qwen/qwen2.5-coder-7b-instruct:free`)

**Inputs required before phase starts:**
- `Proposed_Patches.txt` (Phase 4 output)
- A sandbox copy of the target system (separate directory — never the production path)

**System prompt for Analyst:**
```
You are a verifier. Apply each patch to a temporary copy of the codebase. Run a basic syntax check and logic test on each changed file. For each patch, output a binary PASS or FAIL verdict. For PASS: state what test confirmed it. For FAIL: state exactly what went wrong and what the error message was. Do not suggest fixes. Output only verdicts with evidence.
```

**Output artifact:** `Verification_Report.json`  
**Output location:** `vault/runs/{run_id}/Verification_Report.json`

**Verification_Report.json schema:**
```json
{
  "run_id": "string",
  "cycle": "A",
  "phase": 5,
  "verified_at": "ISO 8601 timestamp",
  "pass_count": "integer",
  "fail_count": "integer",
  "results": [
    {
      "item_id": "string — matches patch item_id",
      "verdict": "PASS | FAIL",
      "evidence": "string — what test confirmed the verdict",
      "metrics_before": {
        "error_count": "integer or null",
        "syntax_errors": "integer or null"
      },
      "metrics_after": {
        "error_count": "integer or null",
        "syntax_errors": "integer or null"
      }
    }
  ]
}
```

**Phase 5 complete when:**
- `Verification_Report.json` exists and is valid JSON
- Every patch from `Proposed_Patches.txt` has a corresponding result entry
- No PASS verdict exists for a patch where metrics_after show regressions vs metrics_before

**Friday escalation trigger:**  
If `fail_count / (pass_count + fail_count) > 0.5`, Friday escalates to Jamie before Phase 6 runs. Jamie decides whether to re-run Surgeon on failed items or proceed to Legacy with partial results.

### Phase 6: Legacy / AAR — Institutional Memory

**What this phase IS:**  
The learning phase. Legacy reads everything that happened in this cycle and distills it into two things: an After Action Review (what happened and why) and at least one new pattern or antipattern to add to the library.

**What this phase IS NOT:**  
Legacy does not propose new fixes. Legacy does not re-evaluate the Heal List. Legacy only captures and generalizes what occurred.

**Model:** `TTLG_LEGACY_A` (fixed: `deepseek/deepseek-r1:free` — cannot be changed by PRIORITY_MODE)

**Inputs required before phase starts:**
- `Heal_List.json`
- `Approved_Heal_List.json`
- `Verification_Report.json`
- All existing patterns from `vault/patterns/`

**System prompt for Legacy:**
```
You are a pattern extractor and historian. You have received a complete record of a diagnostic cycle: what issues were found, which were approved for fixing, and whether those fixes worked. Write an After Action Review that explains: what was misaligned, why it was misaligned, what was changed, and what proof exists that it worked or didn't. Then extract at least one new rule, pattern, or antipattern to add to the pattern library. The pattern must be specific enough to help the next Scout or Thinker detect the same class of problem faster.
```

**Output artifacts:**
- `AAR.json` → `vault/runs/{run_id}/AAR.json`
- `New_Pattern.json` → `vault/patterns/{pattern_id}.json`

**AAR.json schema:**
```json
{
  "run_id": "string",
  "cycle": "A",
  "phase": 6,
  "created_at": "ISO 8601 timestamp",
  "target_system": "string",
  "issues_found": "integer",
  "issues_approved": "integer",
  "patches_passed": "integer",
  "patches_failed": "integer",
  "narrative": "string — plain language summary of what happened",
  "root_causes": ["list of root cause descriptions"],
  "what_changed": ["list of what was actually modified"],
  "proof": "string — what evidence confirms improvement",
  "lessons": ["list of lessons for future cycles"]
}
```

**New_Pattern.json schema:**
```json
{
  "pattern_id": "string — PAT-{YYYYMMDD}-{sequence}",
  "created_from_run": "string — run_id",
  "pattern_type": "pattern | antipattern",
  "name": "string — short descriptive name",
  "description": "string — what to look for",
  "detection_signal": "string — what Scout should flag to trigger this pattern check",
  "constitutional_rule": "string — which Constitution rule this relates to, or null",
  "remediation": "string — what Surgeon should do when this pattern is confirmed",
  "evidence_from_run": "string — specific example from this cycle"
}
```

**Phase 6 complete when:**
- `AAR.json` exists, is valid JSON, and `narrative` field is non-empty
- `New_Pattern.json` exists in `vault/patterns/` with a unique pattern_id
- Both files reference the same `run_id`

## Cycle A Completion — What "Done" Means

A Cycle A run is complete when all of the following exist in `vault/runs/{run_id}/`:

```
Scout_Map.json           ✅ Phase 1
Heal_List.json           ✅ Phase 2
Approved_Heal_List.json  ✅ Phase 3
Proposed_Patches.txt     ✅ Phase 4
Verification_Report.json ✅ Phase 5
AAR.json                 ✅ Phase 6
vault/patterns/PAT-*.json ✅ Phase 6 (new pattern)
run.log                  ✅ Full structured log
```

Friday reads this directory after Phase 6 completes and confirms all artifacts present before marking the run complete in its decision log.

---

## Iteration — How Each Cycle Improves the Next

After every completed Cycle A run:
1. `New_Pattern.json` is added to `vault/patterns/`
2. On the next Scout run, Thinker receives all accumulated patterns
3. Thinker uses patterns to detect the same class of problem faster and with higher confidence
4. The pattern library grows with each client engagement, making TTLG more accurate over time

This is the compounding intelligence loop. It is why TTLG becomes more valuable with each run rather than staying flat.
