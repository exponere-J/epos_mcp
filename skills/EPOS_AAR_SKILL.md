# EPOS AAR Skill

## When to use
Apply at the end of every coding session. Non-negotiable.

## Template
Follow the 9-section template from AAR_CONSTITUTIONAL_AMENDMENT.md.

## The 9 Mandatory Sections
1. **Session Identity** — date, directive, missions planned/completed
2. **Running Systems Inventory** — table with file/imported/running/verified/evidence columns
3. **Planned vs Actual** — for every mission
4. **What Went Well** — specific architectural decisions that worked
5. **What Went Wrong** — at least 1 entry, no session is perfect
6. **What Was Learned** — patterns that apply beyond this session
7. **Doctrine Impact** — any constitutional or framework changes
8. **Ecosystem State Delta** — before/after numbers
9. **Next Session Guidance** — what to build next

## Rules
1. AAR is the FINAL mission — cannot be skipped
2. Must include Running Systems Inventory table
3. Must have at least 1 "What Went Wrong" entry
4. Must include before/after ecosystem state delta
5. Must be PRINTED to terminal before session ends (not just stored)
6. Must be stored at project root AND context_vault/aar/

## Anti-patterns (DO NOT DO)
- Summary tables instead of AARs
- "Everything went well" with no failures listed
- Storing AAR but not displaying it
- Skipping AAR because context limits approach (AAR takes priority)
- Marking missions complete without verification

## The Test
If you cannot show evidence that a system is RUNNING (not just that the file exists), it does not belong in the "Complete" column. List it in "What Went Wrong" with the specific reason.

*1% daily. 37x annually.*
