## 5. What Went Wrong

- **Agent Zero image is ~1.17GB and still downloading.** The directive expected AZ to be deployable quickly, but the cloud image pull is bandwidth-constrained. Container is not yet running. The bridge remains untested against a live AZ engine.
- **browser-use Agent returned `None` result with Ollama.** The local llama3 model does not produce structured output that browser-use expects. Groq (llama-3.3-70b) works better but hits json_schema format issues on some tasks. The browser LAUNCHES and NAVIGATES correctly, but the LLM reasoning layer needs a more capable model (Claude or GPT-4o) for complex multi-step browser tasks.
- **groq package was downgraded** from 1.1.2 to 0.37.1 by browser-use dependency resolution. This may affect groq_router.py if it depends on newer groq API features. Needs verification.
- **Mission 6 (Loose Ends) deferred again.** 9th Order flag parsing, Drive scanner, wake trigger script remain unfixed for the third session.
- **Daemon is not yet a Windows Service.** It runs as a background process that will die when the terminal closes. NSSM wrapping needed for true persistence.

## 6. What Was Learned

- **"Nothing is done until it runs" is the correct standard.** Previous sessions created 30+ files that were never imported, never called, never verified. This session verified every system with actual execution evidence.
- **The Scenario Projection Block works.** Testing 5 environmental assumptions before coding prevented building on broken foundations. This should be mandatory in every directive.
- **browser-use needs a capable LLM for complex tasks.** Simple navigation works with any model. Multi-step form filling and posting requires json_schema support. The fallback chain (BrowserUse -> staging) is the correct architecture — BrowserUse handles what it can, staging catches what it can't.
- **Event reactors transform a log into a command bus.** The moment the reactor started tailing and dispatching, EPOS went from "publishes events that nothing reads" to "publishes events that trigger actions." This is the architectural inflection point.
- **APScheduler BackgroundScheduler is lightweight and immediate.** Zero configuration, in-process, tested in 3 seconds. The correct choice for EPOS's scale.
