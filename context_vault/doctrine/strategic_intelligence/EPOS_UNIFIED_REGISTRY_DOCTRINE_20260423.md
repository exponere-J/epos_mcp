## II. Deep Dive: The MiroFish Simulation Engine

The MiroFish Engine is the "Looking Glass" through which all EPOS
assets must pass. It is the core of our **Recursive Optimization Loop**.

### The 4-Cycle Fortification Process

#### Cycle 1 — Narrative Validation (The Vibe Check)
- **Action:** Simulates 100 virtual buyers to score listing copy on
  "Emotional Resonance" against extracted pain language.
- **Fail Condition:** Average score < 7.0 → Content Lab rewrites using
  deeper CCP-extracted sentiment.

#### Cycle 2 — Environmental Diagnostics (The Tech Attack)
- **Action:** Attacks the product with the **Dependency Ledger (GT-001
  to GT-010)** (e.g., pathing errors, Docker downtime).
- **Fail Condition:** Product lacks a recovery path or clear error
  guidance for known footguns.

#### Cycle 3 — Prescription (Bridge Creation)
- **Action:** Claude Code generates "Bridge Tools" (auto-installers,
  setup scripts, or dynamic FAQs) for every failure identified in Cycle 2.

#### Cycle 4 — Improvement Flush (The Feedback)
- **Action:** All "Bridge Tools" and code hardening are **flushed back
  to the Internal Vault**.
- **Result:** The internal business becomes more resilient as a direct
  side-effect of market-facing development.

### The Improvement Flush as Flywheel

Every product we sell makes EPOS stronger. Every Bridge Tool created
for a customer's environment issue also hardens our own Docker stack.
Every ProseMirror workaround discovered for Gumroad listing automation
also teaches the BrowserUse arm about rich text editors everywhere.
**The marketplace isn't a distraction from the build — it's the
build's most valuable teacher.**
