## THE 7-VECTOR RESEARCH PROTOCOL — HOW IT WAS BUILT

The repeated pattern of workarounds in build sessions pointed
to one root cause: building before the research was complete.
The cost of that pattern was documented in the Architectural
Analysis and Failure Scenarios catalog.

The protocol formalizes what the human mind does naturally
when making high-stakes decisions: play out the scenario
across multiple dimensions before committing.

```
V1 — PRODUCT     What does this do for the user?
V2 — MARKET      Who needs this and why now?
V3 — TECHNICAL   What are the real constraints?
V4 — IMPLEMENTATION  What is the actual build path?
V5 — LAUNCH      How does it reach the first user?
V6 — OPERATIONS  How does it run after it is built?
V7 — LEGAL/SOVEREIGNTY  What could stop this?
```

**The verdict framework**:
- GO: all 7 vectors clear, build queue immediately
- PIVOT: 1-2 vectors need revision, rewrite before building
- KILL: fundamental blocker in V2, V3, or V7
- DEFER: sound idea, waiting for dependency

**The agent matriculation rules** (7 rules):
Every idea that enters the system follows a governed path.
Friday triages daily. RS1 generates briefs. NotebookLM
synthesizes. This thread converts to contracts. CODE builds.
EVL1 learns from outcomes. Each cycle is faster than the last.

**Why this matters**: The research protocol is not a process.
It is a constitutional document. No build enters the pipeline
without it. This single protocol prevents 80% of the rework
that has characterized previous sessions.

## THE GOOGLE SHEETS ARCHITECTURE INSIGHT

The conversation about Sheets as CRM, PM, and CMS database
led to an architectural breakthrough: the Sheet is not a
spreadsheet. It is a command surface with embedded execution
triggers.

**Three control vectors identified**:

Vector 1 — Apps Script (embedded in the sheet)
  Custom menus, buttons, onEdit triggers, doPost web apps
  Every row can have a [▶ Distribute] button that fires
  a POST to EPOS FastAPI which triggers the content loop

Vector 2 — Google CLI + Sheets API v4 + gspread
  Friday reads My Work Today tab at 07:15 AM briefing
  Friday writes TTLG scores back after diagnostics run
  All without opening a browser — pure API calls

Vector 3 — HYPERLINK as execution trigger
  Human-initiated: Jamie clicks → doGet → EPOS dispatch
  Agent-initiated: Friday POSTs directly to web app endpoint
  Two different call patterns for two different actors

**The Perplexity research confirmed** (locked decisions):
- Simple onEdit = UI feedback only
- Installable onEdit = any agent-facing trigger
- Supabase = source of truth, last_modified wins
- HMAC-SHA256 on all inbound webhooks
- Batch everything — 300 read/write per minute quota
- No whole-column conditional formatting
- Execute as Me + UUID shared secret in ScriptProperties

that shipped.
